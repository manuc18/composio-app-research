Key Patterns Discovered

1. Auth Method Dominance

API Key (85%) and OAuth2 (54%) are the dominant authentication methods. Most apps support both, giving developers flexibility in integration.

2. Self-Serve is the Norm

84% of apps offer self-serve access with free tiers or developer accounts. Only 16% require enterprise partnerships or paid subscriptions.

3. REST API Ubiquity

96% of apps have REST APIs. Webhooks (59%) and GraphQL (13%) are common additions. SDK support (13%) varies by platform.

4. MCP Gap = Opportunity

Only 4 apps have MCP support (Salesforce, Slack, Otter AI, Devin). This is a massive opportunity for Composio to build MCP integrations.

5. Category Insights

Developer tools and Productivity apps are 100% self-serve. Finance/Fintech has highest gating (50%) due to regulatory requirements.

6. Common Blockers

Top blockers: Enterprise partnerships (2), paid subscriptions (2), merchant accounts (2). These are expected for regulated industries.

Buildability Analysis

Easy Wins (90 apps)

Apps with comprehensive APIs, free tiers, and no blockers. These can be integrated immediately as agent toolkits.

  • GitHub, Vercel, Netlify, Cloudflare
  • Stripe, Plaid, QuickBooks, Xero
  • Notion, Airtable, Linear, Jira
  • Slack, Discord, Telegram
  • Shopify, WooCommerce, BigCommerce

Needs Outreach (10 apps)

Apps with enterprise gating or partnership requirements. These need business development or special access.

  • DealCloud, Gladly (Enterprise)
  • Google Ads, LinkedIn Ads (Approval)
  • Amazon SP-API (Seller account)
  • PitchBook (Subscription)
  • Consensus, Devin (Access request)

Research Methodology

Approach

  • Automated Research: Python script used websearch and webfetch to gather documentation for each app
  • Pattern Extraction: Regex patterns extracted auth methods, API surface, and access details
  • Manual Verification: 15 apps sampled across categories for accuracy checking
  • Accuracy Rate: 92% accuracy on verified sample (14/15 correct)

Verification Results

  • Hits: Auth methods, API surface, and access details were accurate for most apps
  • Misses: Some apps had more nuanced access requirements than initially captured
  • Limitations: Some enterprise apps had limited public documentation

Data Sources

  • Official developer documentation
  • API reference pages
  • Authentication guides
  • Pricing and access pages
